- バージョン：1.0（PoC検証版）
- 作成日：2026年4月
- ステータス：PoC検証フェーズ

## 1. 背景・目的

### 1.1 背景

製造業の購買部門では、仕入先から原材料の値上げ要請を受けた際に、その妥当性を判断するために多くの情報を収集・分析する必要がある。現状では以下の課題がある:

- 品質仕様書（PDF）、価格交渉記録（PDF）、単価改定履歴（CSV/Excel）など、情報が複数の非構造化ドキュメントに散在している
- 原材料の成分構成・市況価格・燃動力費・物流費など、多数の変動要因を手作業で集約・計算している
- 過去の交渉履歴や市況トレンドの参照に多大な工数がかかっている
- 値上げの妥当性を客観的に評価する仕組みがない

### 1.2 目的

本 PoC の目的は以下を検証することである。

1. **非構造化ドキュメントからの構造化データ自動抽出**: Gemini のマルチモーダル LLM を使い、品質仕様書・価格交渉記録等の PDF から、成分情報・価格・物流情報などを構造化データとして自動抽出できるか
2. **原価シミュレーション**: 抽出データと市況データを組み合わせ、原材料の推定原価を算出し、仕入価格の妥当性を可視化できるか
3. **AI による交渉支援**: 値上げ理由の妥当性分析や関連情報の要約を AI が提供できるか

### 1.3 期待される効果

- 価格交渉の準備工数の大幅削減
- データに基づく客観的な妥当性判断
- 過去の交渉履歴・市況トレンドの即時参照
- 値上げ要請に対する交渉力の向上

## 2. 検証スコープ

### 2.1 PoC で検証すること

| 項目         | 内容                                                                 |
| ------------ | -------------------------------------------------------------------- |
| 検証対象     | 品質仕様書・価格交渉記録等の PDF からの構造化データ抽出精度           |
| テストデータ | WK-680RIP 965, GW-33126 等の既存書類をサンプルとして使用               |
| 主要機能     | ドキュメント取込・データ抽出・原価シミュレーション・妥当性分析         |
| 使用モデル   | Gemini 2.5 Pro（マルチモーダル LLM）                                  |
| データ管理   | 構造化 CSV（ローカル）                                                |
| 検証工数     | 10〜20 時間程度                                                       |

### 2.2 PoC に含めないこと

- 市況データの自動スクレイピング・定期取得（手動入力または静的データで代替）
- 本番環境へのデプロイ・インフラ構築
- ユーザー認証・権限管理
- 基幹システム（TSFrame 等）との連携
- 大量品目（100 品目以上）での性能検証

## 3. システム概要

### 3.1 対象ドキュメントと抽出情報

| ドキュメント種別     | 形式      | 抽出する情報                                                     |
| -------------------- | --------- | ---------------------------------------------------------------- |
| 品質仕様書           | PDF       | 品名、文書番号、分類番号、成分構成（名称・含有量）、用途、納入者、製造者、物流情報（経路・所要時間・運送会社） |
| 価格交渉記録         | PDF       | 改定前価格、改定後価格、改定時期、値上げ理由、交渉経緯、見積書情報 |
| 単価改定履歴         | CSV/Excel | 品目コード、品目名、荷姿、発効日、単価、仕入先、メーカー     |

### 3.2 処理フロー

**Step 1: ドキュメントからの構造化データ抽出**

| ステップ | 処理内容                                          | 使用技術              |
| -------- | ------------------------------------------------- | --------------------- |
| 1        | PDF/CSV をアップロード                              | Python ファイル読込    |
| 2        | Gemini LLM に PDF を送信し構造化データを抽出         | Gemini 2.5 Pro        |
| 3        | 抽出結果をスキーマに沿って CSV に格納               | Python データ処理      |
| 4        | 抽出精度を画面上で確認・手動修正                    | Streamlit UI          |

**Step 2: 原価シミュレーション**

| ステップ | 処理内容                                          | 使用技術              |
| -------- | ------------------------------------------------- | --------------------- |
| 1        | 品目を選択                                         | Streamlit UI          |
| 2        | 成分別の市況単価×含有量で材料費を算出               | Python 計算           |
| 3        | 燃動力費・物流費・労務費を加算し推定原価を算出       | Python 計算           |
| 4        | 推定原価 vs 実際の仕入価格を比較表示                | Streamlit UI          |

**Step 3: AI による交渉支援**

| ステップ | 処理内容                                          | 使用技術              |
| -------- | ------------------------------------------------- | --------------------- |
| 1        | 値上げ理由と市況データを AI に入力                  | Gemini 2.5 Pro        |
| 2        | 値上げの妥当性を分析・コメント生成                  | Gemini 2.5 Pro        |
| 3        | 分析結果を画面に表示                                | Streamlit UI          |

### 4.1 統合 CSV スキーマ

| # | カテゴリ       | フィールド                                          |
|---|---------------|------------------------------------------------------|
| 1 | 品目情報       | 文書番号、品名、分類番号（大/中/小/商標）             |
| 2 | 物流情報       | 運送経路、所要時間、運送会社、輸送距離                 |
| 3 | 成分構成       | 成分名×10、含有量×10                                  |
| 4 | 材料構成       | 材料名×10、含有量×10                                  |
| 5 | 価格改定       | 元価格、改定後価格、改定時期、改定理由               |
| 6 | 市況単価       | 成分1〜4の市況単価、市況情報URL                       |
| 7 | 燃動力費       | 燃料種別（LNG/電気等）×4、市況情報URL、単価           |
| 8 | 労務費         | 時間あたり労務費、市況情報URL                         |
| 9 | 運賃           | 円/m 単価                                              |

参照: `Pocのゴール/260401_当社がやりたい事のイメージメモあり.xlsx - P. 04データイメージ.csv`

| 文書番号・品名・分類    | 品質仕様書 PDF           | Gemini LLM 抽出     |
| 成分構成・含有量       | 品質仕様書 PDF           | Gemini LLM 抽出     |
| 物流情報               | 品質仕様書 PDF           | Gemini LLM 抽出     |
| 価格改定情報           | 価格交渉記録 PDF         | Gemini LLM 抽出     |

### 4.3 サンプルデータ（WK-680RIP 965）との突き合わせ結果

サンプルデータ3ファイルから統合CSVの各カラムへ転記可能かを調査した結果。

#### 転記可能な項目（サンプルデータから自動抽出可能）

| カラム | 転記元 | 備考 |
|--------|--------|------|
| 文書番号 | 品質仕様書 P1 | `1203010160` |
| 品名 | 品質仕様書 P1 | `WK-680RIP (965)` |
| 分類番号（大/中/小/商標） | 品質仕様書 P1 | `12/03/01/0160` |
| 運送経路 | 品質仕様書 P3 輸送方法欄 | `静岡県富士市→貴社土浦事業所` |
| 所要時間 | 品質仕様書 P3 輸送方法欄 | `6時間前後` |
| 運送会社 | 品質仕様書 P3 輸送方法欄 | `株式会社タカキュー、英運輸送株式会社` |
| 成分名 | 品質仕様書 P1 化学名又は組成欄 | `セルロースパルプ、ケイ酸塩鉱物、二酸化チタン`（名称のみ） |
| 元価格・改定後価格 | 単価改定CSV + 価格交渉記録 | `15.17→16.3→19.56` |
| 改定時期 | 単価改定CSV 発効日 | `2021/5/1`, `2022/4/1`, `2022/11/1` |
| 改定理由 | 価格交渉記録 | `為替及びユーティリティ費用の上昇`、`パルプ価格高騰（190%）` 等 |
| 燃動力の種別 | 価格交渉記録 | LNG・天然ガスへの言及あり（ただし分類名としてではなく文脈中） |

#### 転記不可能な項目（サンプルデータに記載なし — 外部データが必要）

| カラム | 不足の理由 | 想定される入手先 |
|--------|-----------|-----------------|
| 輸送距離（km） | 品質仕様書に経路・所要時間はあるがkm数の記載なし | Google Maps 等で算出、または手動入力 |
| 成分の含有量（%） | 品質仕様書には成分名のみ（`セルロースパルプ、ケイ酸塩鉱物、二酸化チタン`）。ゴールCSVにある `73〜83%` 等の詳細割合は記載なし | SDS（安全データシート）等の別資料 |
| 成分1〜4の市況単価 | 外部市況データ。社内ドキュメントには含まれない | 外部市況サイト（gdfreak.com, 経産省統計等） |
| 成分1〜4の市況情報URL | 同上 | 手動設定 |
| 燃動力の単価（円） | 価格交渉記録に「天然ガス高騰」の言及はあるが具体的な単価はなし | 外部市況サイト |
| 燃動力の市況情報URL | 同上 | 手動設定 |
| 労務費（時間当り） | サンプルデータのどこにも記載なし | 外部統計データ（厚労省 賃金構造基本統計等） |
| 労務費の市況情報URL | 同上 | 手動設定 |
| 運賃（円/m） | サンプルデータのどこにも記載なし | 運送会社見積もり、またはトラック協会統計等から算出 |

#### 結論

統合CSVの約半分（品目基本情報・物流経路・成分名・価格改定履歴）はサンプルデータから自動抽出可能。残り半分（市況単価・燃動力費・労務費・運賃）は外部データソースからの取得が必要であり、PoCでは手動入力で対応する。成分の含有量（%）については、品質仕様書には成分名しか記載されておらず、SDS等の別資料が必要。

## 6. 技術スタック

| レイヤー         | 技術                | 備考                                         |
| ---------------- | ------------------- | -------------------------------------------- |

### 6.1 PoC 実装方針

- Web UI は Streamlit 単体で構築し、起動コマンドは `streamlit run app.py` を想定する
- 統合データの保存先は `data/extracted/` とする
- アップロードドキュメントの保存先は `data/uploads/` とする
- API キーは `.env` に `GEMINI_API_KEY` として設定する
- シミュレーター画面の参照 UI は `Pocのゴール/260401_当社がやりたい事のイメージメモあり.xlsx - P. 05シミュレーター.pdf` を参照

## 10. 制約・前提条件

### 10.1 制約事項

- Gemini 2.5 Pro のマルチモーダル入力には API の制約（トークン上限等）がある
- PDF の品質（スキャン解像度、手書き注記等）により抽出精度が変動する可能性がある
- 市況データは PoC では手動入力とし、自動取得は後フェーズとする
- 無料枠利用時、入力データが Google のモデル改善に使用される可能性がある（機密データは有料プランを使用すること）

### 10.2 前提条件

- Google AI Studio にて API キーを取得済みであること
- Python 3.11 以上の開発環境が整っていること
- 検証用ドキュメント（品質仕様書・価格交渉記録）が用意できること
- インターネット接続環境があること（Gemini API はクラウド呼び出し）
